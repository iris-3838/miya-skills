# サブスクリプション検出アプローチ

## 問題

PayPay CSVとPayPayカードの二重取り込みにより、Zaim内の取引が **高頻度で重複** する可能性がある。
これを考慮せずにサブスク検出を行うと、実際より多く出現回数が見え誤った判断を招く。

## 解決策: 3段階アプローチ

### 1. 重複除去（Dedup）

同一の取引が PayPay CSV（残高支払い）と PayPayカード請求の両方で登録されている。
同じ `日付 + 金額 + 店舗名` の組み合わせは1件とみなす。

```python
seen = set()
for r in rows:
    key = (r["date"], r["amount"], r["place"])
    if key not in seen:
        seen.add(key)
        deduped.append(r)
```

**結果**: 重複除去により取引数が大幅に減少する（具体的な割合はデータによる）。

### 2. 全カテゴリスキャン

旧来の方法は特定カテゴリ（通信・エンタメ・医療保険・その他）のみを対象としていたが、
食費や日用雑貨にもサブスク（Amazon定期便など）が混在する可能性があるため、
**全カテゴリ**を対象にスキャンする。

### 3. 価格改定検出

同一店舗で金額の異なるレコードをグループ化し、プラン変更や値上げを検出。

## 検出ルール

| 条件 | 分類 |
|------|------|
| 2ヶ月以上 + サービス名キーワード一致 + ¥100以上 | 確定サブスク |
| 3ヶ月以上 + 各月1回程度 + ¥300以上 | 確定サブスク |
| 2ヶ月のみ + 各月1回 + ¥300以上 | 要確認 |
| 2ヶ月以上 + ¥500以上（複数回/月あり） | 参考パターン |

## 重複調査テクニック

重複が疑われた場合の調査手順:

1. **ID範囲を確認**: 全取引のIDを昇順に並べ、ギャップを確認する。連続したID範囲が2つある場合（例: xxxxxx1xxx と xxxxxx2xxx）、それはバッチインポートの痕跡。
2. **ペアを検証**: 重複ペアを特定したら、`comment` フィールドの違いを確認（一方が"クレジット"、他方が"マネーライト"なら、内訳が異なる同一取引）。
3. **IDと日付の相関**: 日付順でIDがジグザグにならないことを確認（IDのある範囲内で日付が混在していたら、それは重複ではなく別々の取引）。
4. **削除戦略**: 基本的に**大きいIDを削除**（新しくインポートされた方が重複コピー）。--keep-older フラグで逆も可能。

Zaim API は `created_at` や `updated_at` を公開していないため、ID順序が唯一の信頼できる判断基準。
