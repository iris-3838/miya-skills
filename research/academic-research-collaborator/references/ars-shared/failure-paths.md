# ARC Failure Paths

ARS deep-researchの失敗パス12シナリオをARC向けに適応した回復戦略。

## 1. RQが曖昧で探索不能

**症状**: ユーザーの関心が広範で、探索可能なquestion sliceに分解できない。

**回復**:
- Socratic対話を継続し、少なくとも概念層（層3）まで具体化する
- 「何を知りたいか」より「何が分かれば次の判断ができるか」に焦点を移す
- それでも絞れない場合は、quick-scanで調査的范围探索を提案

## 2. 探索結果が0件

**症状**: 全sourceが `zero_hits` を返した。

**回復**:
- クエリの言語・表記ゆれ・同義語を確認
- source選定の妥当性を再検討（DBの分野適合性）
- seed文献のDOIがあれば引用探索へ切り替え
- そもそも研究されていない領域の可能性を人間に報告

## 3. 全候補がpaywallでアクセス不能

**症状**: OA文献が0件で、全候補が `acquisition_required`。

**回復**:
- acquisition queueに全件登録
- `acquisition_pending` に遷移
- 機関アクセス・ILL・著者プレプリント・ResearchGateの経路を明示
- 取得不能の場合の研究戦略（抄録ベースのscoping review等）を提案

## 4. API rate limit / 予算枯渇

**症状**: OpenAlex等が429を返し続ける。

**回復**:
- retry方針（待機時間・日付変更）を人間に確認
- 無料source（Crossref Polite）への切り替えを提案
- 既存のsource run結果だけで分析可能か判断
- 不足する場合はacquisition pendingとして記録

## 5. 候補が多すぎて処理不能

**症状**: 1episodeで数十〜数百件の候補が発生。

**回復**:
- 抄録スクリーニングで第一段階の絞り込み
- 大量要約を生成しない
- episodeを日付・source・概念で分割し、人間が優先順位を決める

## 6. 重複・既知文献ばかり

**症状**: known_item_recallが高く、新規発見がほとんどない。

**回復**:
- 既知文献の引用ネットワーク（前方・後方）から新規entry pointを探す
- 隣接分野・異なる学派の語彙で再検索
- 言語・地域を変えた探索を提案
- 「探索済み」として経路を閉じる判断は人間が行う

## 7. Devil's Advocateが深刻な欠陥を指摘

**症状**: DAがRQの前提・方法・範囲に修正が必要な問題を発見。

**回復**:
- DA指摘をRQ briefに追記
- 軽微な修正 → RQを更新してplanを続行
- 重大な修正 → designに戻ってRQを再設計
- DA指摘を棄却する場合は理由を記録

## 8. 反証文献が見つからない

**症状**: 対立立場・批判・代替枠組みを示す文献が発見されない。

**回復**:
- 反証route（A5, B5, B6）が未着手でないか確認
- 隣接分野の批判的レビュー論文を探す
- 「反証不在」自体が知見であることを明記
- 反証探索を諦める判断は人間が行う

## 9. サブエージェント間の矛盾

**症状**: delegate_taskで並列実行したsource間で結果の質・量・解釈に大きな差がある。

**回復**:
- source run manifestのstatusとerror_detailを比較
- 矛盾を解消しようとせず、差異をそのまま人間に報告
- agentが一方の結果を「正」と判断しない

## 10. manifest破損・不正な状態

**症状**: YAMLパースエラー、不正なphase名、revision不一致。

**回復**:
- 自動修復しない
- `.tmp` ファイルがあれば前回の中断を示唆
- 人間に破損内容を報告し、手動修復またはbackupからの復元を提案
- 復旧までphase遷移をブロック

## 11. セッション中断・再開の不整合

**症状**: 中断前のphaseと再開時のartifactの間に矛盾がある。

**回復**:
- manifestの `state.current_phase` と `completed_phases` を読む
- 対応するartifactの存在を確認
- phaseとartifactが一致しない場合は前のphaseに差し戻す
- Kanban taskがある場合はrevision一致を検証

## 12. 研究スコープのドリフト

**症状**: 探索を重ねるうちにRQから離れた方向へ拡がっている。

**回復**:
- 各reflectionで元のRQと現在の探索方向を照合
- ドリフトが意図的（人間が指示）ならRQを更新
- ドリフトが非意図的なら元のRQに戻るか、分岐を明示的に記録
- reflection reportで「RQとの関係」を常に明記
